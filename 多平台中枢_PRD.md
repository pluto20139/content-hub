# PRD：多平台内容中枢（Content Hub）

---

## 1. 概要（Summary）

本文档定义了一套"配置驱动抓取"的多平台内容聚合系统。系统由**配置管理端**、**后端自动化引擎**和**用户端 H5**三个模块组成，帮助个人用户在一个页面内集中追踪抖音、B站、知乎、YouTube 四个平台上关注的博主最新内容，并一键跳转至原生 App 查看。

> **MVP 阶段范围锁定**（详见 SPEC v1.5）：
> - **平台范围**：MVP 仅支持 B站、YouTube、知乎三个平台。抖音（Douyin）因需代理 IP + 短链解析，技术风险较高，推迟至 Phase 2。
> - **告警静默期**：MVP 不实现 F9.3 告警静默期（24h 内不重复告警），每次达到条件（fail_count ≥ 3）均触发告警。
> - **知乎 Deep Link**：MVP 不做知乎 Deep Link 跳转，统一使用 `original_url` 在浏览器中打开。B站和 YouTube 保留 Deep Link。
> - **活跃度分级提示**：MVP 简化为统一文案"⚪ 该博主已超过 {N} 天未更新"，不做 30 天/90 天分级阈值。

## 3. 背景（Background）

### 上下文

当前主流内容平台（抖音、B站、知乎、YouTube）各自拥有独立的推荐算法和信息流，用户关注同一批创作者时，需要在 4 个 App 之间反复切换才能看完所有更新。没有一个轻量工具能让用户"在一个页面里看完所有平台的内容更新"。

### 为什么是现在

- 四大平台均已开放了不同程度的公开主页/专栏页面，使得无需登录的增量抓取成为可能。
- Deep Link 协议（`snssdk1128://`、`bilibili://`、`zhihu://`）在主流手机上已广泛支持，H5 跳转原生 App 体验成熟。
- RSSHub 等开源中转方案为反爬严格的平台（如知乎）提供了可行替代路径。

### 核心思路

**不造内容仓库，只做智能中转。** 系统不存储视频/文章正文，只保存"信息卡片"（标题、封面、链接），所有阅读行为回到原平台完成。这让系统极轻量，同时尊重内容平台的版权。

## 4. 目标（Objective）

### 产品目标

构建一个极简的个人内容聚合工具，让用户通过一个 H5 页面即可浏览所有关注博主在四个平台上的最新内容，并一键跳转至原生 App 深度阅读/观看。

### 对齐公司/个人战略

- 降低日常内容追踪的时间成本，从"打开 4 个 App 逐个刷"变为"一个页面全看完"。
- 为后续扩展更多平台（小红书、Twitter/X、微信公众号）建立可扩展架构。

### 关键成果（Key Results）

| KR | 衡量指标 | 目标值 |
|---|---|---|
| KR1：内容覆盖完整度 | 已开启监控的博主，新内容在 1 小时内出现在 H5 信息流的比例 | ≥ 95% |
| KR2：用户触达效率 | 用户点击卡片后成功跳转到原生 App 的比例 | ≥ 80% |
| KR3：系统稳定性 | 单月因 Cookie 失效或抓取异常导致的信息流断更次数 | ≤ 2 次 |
| KR4：数据效率 | 数据库平均单条记录大小 | ≈ 1 KB（上限 3 KB） |

## 5. 目标用户（Market Segments）

### 主要用户画像

**"多平台重度内容消费者"**：日常同时关注抖音、B站、知乎、YouTube 上的 20~100 位创作者，每天花 15~60 分钟在不同 App 之间切换查看更新。他们的核心痛点不是"找不到好内容"，而是"好内容散落在太多地方，追踪太累"。

### 次要用户画像

**"个人/小团队运营者"**：需要监控竞品或行业 KOL 在多个平台上的发布动态，要求轻量、低成本、无需复杂部署。

### 约束条件

- 用户场景以**手机端 H5** 为主（可能在微信内置浏览器中打开），因此 Deep Link 需兼容微信等受限环境。
- 系统面向**个人或小团队使用**，不涉及多租户、权限分级等企业级需求。

## 7. 解决方案（Solution）

### 7.1 用户体验与交互流程

#### 配置管理端（管理员使用）

```
用户操作流程：
1. 打开配置管理页面
2. 在顶部通用输入框粘贴博主主页链接（如 https://v.douyin.com/xxx/）
3. 系统自动识别平台 → 解析出唯一标识 → 存入数据库
4. 博主出现在监控列表中，默认状态为"开启"
5. 管理员可随时切换开关（is_active）来开启/关闭监控
6. 列表中展示每个博主的"最后同步时间"和"运行状态"
```

#### 用户端 H5（日常浏览使用）

```
用户操作流程：
1. 打开手机浏览器/微信访问 H5 页面
2. 看到按时间倒序排列的信息卡片流
3. 每张卡片带有平台专属彩色 Tag（粉色B站、红色YouTube、黑色抖音、蓝色知乎）
4. 点击某张卡片 → 系统尝试 Deep Link 唤醒原生 App
5. 若唤醒失败（如在微信内）→ 弹出兜底弹窗：
   "链接已自动复制，打开 App 即可直接看" + "点击此处网页打开"
```

### 7.2 核心功能详述

#### 模块一：配置管理端（Configuration Backend）

**功能 F1：统一添加入口**

- 页面顶部提供通用输入框，支持粘贴任意平台博主主页/专栏链接
- 输入后点击"添加"，前端校验 URL 格式合法性，调用后端解析接口

**功能 F2：平台自动识别与解析**

后端根据 URL 特征自动识别平台并提取核心标识：

| 平台 | 输入示例 | 解析的核心唯一标识 | 技术前置条件 |
|---|---|---|---|
| **抖音 (Douyin)** | `https://v.douyin.com/xxx/` | `sec_uid`（短链重定向后从网页 URL 提取） | 无需登录，需稳定代理 IP |
| **哔哩哔哩 (Bilibili)** | `space.bilibili.com/uid` | `mid`（用户唯一空间 ID） | 需配置并定期更新 B 站账号 Cookie |
| **知乎 (Zhihu)** | `zhihu.com/people/id` | `people_id` 或 `column_id` | 反爬极严，需动态 Cookie 或 RSSHub 中转 |
| **YouTube** | `youtube.com/@channel_name` | `channelId`（官方 API search 接口转换） | 服务器需海外网络访问能力 + 官方 API Key |

- 解析成功后，将 `{平台名, 唯一标识, 原始URL, is_active=true}` 写入 `monitors` 表
- 解析失败时，返回明确错误提示（如"无法识别该平台"或"链接格式不正确"）

**功能 F2-补充：博主昵称获取策略**

- **优先获取**：添加时同步调用平台接口获取博主昵称（如 B 站用户信息 API、YouTube channels API）
- **降级策略**：若获取失败（网络异常、Cookie 过期、API 限流等），使用默认名称格式：**`{平台中文名}_{native_id前8位}`**（如"B站_mid12345"、"抖音_sec98765"）
- **异步补充**：昵称获取不作为添加流程的阻塞步骤。先快速入库（使用降级名称），后台 Cron 在下一轮抓取时顺带刷新昵称
- **手动编辑**：配置管理端支持管理员点击昵称字段直接修改（行内编辑），修改后系统不再自动覆盖

**功能 F3：监控状态控制**

- 列表展示所有已添加的博主，每行包含：博主名称、平台标签、开关按钮
- 一键切换 `is_active` 字段，开启时 Cron Job 纳入轮询，关闭时跳过

**功能 F4：状态维护与反馈**

- 每个博主行显示"最后同步时间"（如 `2026-06-20 14:00`）
- 运行状态标识三种：
  - 🟢 **正常**：最近一次抓取成功
  - 🟡 **Cookie 过期**：连续失败 1~2 次
  - 🔴 **接口受限**：连续失败 ≥ 3 次，已触发告警
- **博主活跃度提示**（独立于运行状态，额外展示）：
  - **NULL 保护**：若 `last_content_at` 为 NULL（理论上不应出现，但作为防御性兜底），不显示任何停更提示
  - 若 `last_content_at` 距今超过 **30 天**，在博主行显示灰色提示："⚪ 该博主已超过 N 天未更新"
  - 若超过 **90 天**，提示升级为："建议关闭监控或移除"
  - 该提示不影响运行状态标识（🟢🟡🔴），两者独立展示

#### 模块二：后端自动化引擎（Backend Automations）

**功能 F5：定时轮询机制（Cron Job）**

```
每 30 分钟执行一次：
├── 【互斥检查】若上一轮仍在运行 → 跳过本轮，记录日志"上一轮未完成，跳过"
├── 获取互斥锁，标记本轮开始
├── 查询 monitors 表中 is_active = true 的所有记录
├── 按 platform 分组，组内串行遍历每条 monitor 记录
│   ├── 抖音适配器：通过代理 IP 请求博主主页，解析最新内容
│   ├── B站适配器：携带 Cookie 请求空间 API
│   ├── 知乎适配器：通过动态 Cookie 或 RSSHub API 获取
│   └── YouTube适配器：调用官方 Data API v3
│       ⚠️ YouTube 专属降频：仅当距该 monitor 上次抓取 ≥ 4 小时时才执行
│       （保护免费 API 配额，10,000 units/天 ÷ 6次/天 ≈ 充足）
│   同一平台内相邻两次请求间隔 ≥ 1.5 秒（防止触发平台频率限制）
├── 每个适配器返回最新 前 6 条内容（增量抓取，穿透置顶内容）
├── 进入数据清洗与去重流程
└── 释放互斥锁，标记本轮结束
```

- **增量抓取策略**：每次请求博主主页前 6 条内容（覆盖置顶帖区域），不翻阅历史，极大降低被封禁风险
- **失败计数**：每条 monitor 记录维护 `fail_count` 字段，请求失败 +1；**请求成功即归零**（与新内容入库无关）
- **并发安全机制**：
  - **Cron 互斥锁**：基于数据库行锁或 Redis 分布式锁，确保同一时刻只有一个 Cron 实例在运行；上一轮未完成则跳过本轮
  - **同平台限速**：同一平台的请求之间强制间隔 ≥ 1.5 秒，避免密集请求触发反爬封禁
  - **写回前校验**：Cron 写回 monitor 状态前，先校验该 monitor 是否仍存在且 `is_active = true`，若已被删除或关闭则跳过写回

**功能 F6：数据清洗与模型统一**

将四个平台不同的原始数据结构，统一裁剪为标准信息卡片模型：

```json
{
  "platform": "bilibili",        // 平台标识
  "native_id": "BV1xx411c7mD",   // 平台内原生内容 ID
  "content_type": "video",       // 内容类型（video/article/question/answer/post）
  "title": "视频标题",
  "cover_url": "https://...",    // 封面图链接
  "original_url": "https://...", // 原文链接
  "published_at": "2026-06-20T10:00:00Z",
  "monitor_id": 42               // 关联的博主监控 ID
}
```

**功能 F7：全局数据去重与元数据同步（UPSERT）**

- 唯一索引：`(platform, native_id)`
- **UPSERT 机制**（取代原 INSERT IGNORE）：
  - **新内容**（native_id 不存在）：INSERT 写入 `contents` 表，`is_display = true`
  - **已存在内容**（native_id 已存在）：UPDATE 覆盖 `title`、`cover_url`、`original_url`，防止死链与破损图片
  - **防复活保护**：UPSERT 命中已有记录时，若该记录 `is_display = false`（已软删除），仅更新标题和图片链接，**严禁重置** `is_display` 状态和 `created_at` 时间，阻断旧数据复活的死循环
- 状态回写（在 Step 6 统一执行）：更新 monitor 的 `last_sync_at`；有新内容入库时额外更新 `last_content_at`

**功能 F8：数据生命周期管理（软删除）**

- **保留策略**：H5 信息流仅展示最近 **30 天**内的内容；超期数据不物理删除，转为软删除保留历史轨迹
- **软删除机制**：每日凌晨执行一次扫描任务，将 `created_at` 超过 30 天且 `is_display = true` 的记录标记为 `is_display = false`
- **软删除 SQL**：`UPDATE contents SET is_display = false WHERE created_at < NOW() - INTERVAL '30 days' AND is_display = true`
- **前端过滤**：H5 信息流接口强制添加 `WHERE is_display = true` 条件，确保软删除内容在用户侧不可见
- **数据价值**：历史数据完整保留，为未来的发文频率分析、博主活跃度挖掘、数据回溯预留空间
- **执行记录**：每次扫描后记录日志（标记条数、执行耗时）

#### 用户端 H5 展现层（User Interface）

**功能 F10：聚合信息流（主页）**

- 按 `published_at` 时间倒序展示所有内容卡片（仅查询 `is_display = true` 的记录）
- 支持无限滚动（分页加载，每页 20 条）
- 每条卡片展示：封面图、标题、平台彩色 Tag、发布时间
- 平台 Tag 配色规范：
  - 抖音：黑色（`#000000`）
  - B站：粉色（`#FB7299`）
  - 知乎：蓝色（`#0066FF`）
  - YouTube：红色（`#FF0000`）
- 支持按平台筛选（顶栏 Tab 切换）

**功能 F11：一键触达跳转（核心交互）**

点击卡片时按优先级执行以下流程：

```
用户点击卡片
  ├── 0. 环境检测：判断当前运行环境（见附录 B.4 详细逻辑）
  │     ├── 微信/支付宝内 → 直接走"复制链接 + 引导在浏览器打开"路径
  │     └── 系统浏览器 → 继续 Deep Link 流程
  │
  ├── 1. 静默复制：Clipboard API 将 original_url 写入系统剪贴板
  │
  ├── 2. Deep Link 唤醒：根据 (platform, content_type) 组合选择 Schema
  │     │
  │     │ 抖音 (douyin)
  │     │   └── video → snssdk1128://aweme/detail/{native_id}
  │     │
  │     │ B站 (bilibili)
  │     │   ├── video   → bilibili://video/{native_id}
  │     │   └── article → bilibili://article/{native_id}
  │     │
  │     │ 知乎 (zhihu)
  │     │   ├── question → zhihu://question/{native_id}
  │     │   ├── answer   → zhihu://answer/{native_id}
  │     │   └── article  → zhihu://zhuanlan/{native_id}
  │     │
  │     │ YouTube
  │     │   └── video → youtube://watch?v={native_id}
  │     │
  │     └── 无法匹配 → 直接使用 original_url 打开网页
  │
  └── 3. 兜底弹窗（若唤醒超时 > 2s）：
        "链接已自动复制，打开 App 即可直接看"
        + [点击此处网页打开] 按钮（window.open original_url）
```

### 7.3 技术架构（Technology）

#### 整体架构

```
┌─────────────────────┐
│   配置管理端 (Web)    │ ← 管理员使用
│  Vue/React + API     │
└─────────┬───────────┘
          │ HTTP API
┌─────────▼───────────┐
│   后端服务 (Node/Go)  │
│  ├── REST API        │
│  ├── 平台适配器层     │
│  ├── Cron Job (30min)│
│  └── 告警模块        │
└─────────┬───────────┘
          │
┌─────────▼───────────┐
│   数据库 (PostgreSQL) │
│  ├── monitors 表     │
│  └── contents 表     │
└─────────────────────┘
          │ HTTP API
┌─────────▼───────────┐
│   用户端 H5 (Mobile) │ ← 日常浏览
│  Vue/React SPA       │
└─────────────────────┘
```

#### 数据模型

**monitors 表**

| 字段 | 类型 | 说明 |
|---|---|---|
| id | INT (PK) | 自增主键 |
| platform | VARCHAR(20) | 平台标识（douyin/bilibili/zhihu/youtube） |
| native_id | VARCHAR(200) | 博主在平台内的唯一标识 |
| display_name | VARCHAR(100) | 博主显示名称（手动修改后不再自动覆盖） |
| name_auto | BOOLEAN | 名称是否为系统自动获取（true=自动，false=管理员手动修改） |
| original_url | VARCHAR(500) | 原始粘贴的链接 |
| is_active | BOOLEAN | 是否开启监控 |
| last_sync_at | TIMESTAMP | 最后成功同步时间（API 请求成功，不代表有新内容） |
| last_content_at | TIMESTAMP | 最后获得新内容的时间（初始值 = created_at；用于判断博主是否仍在活跃更新） |
| fail_count | INT | 连续失败次数 |
| status | ENUM | normal / cookie_expired / rate_limited |
| created_at | TIMESTAMP | 添加时间 |

**contents 表**

| 字段 | 类型 | 说明 |
|---|---|---|
| id | INT (PK) | 自增主键 |
| platform | VARCHAR(20) | 平台标识 |
| native_id | VARCHAR(200) | 内容在平台内的原生 ID |
| content_type | VARCHAR(20) | 内容类型（video / article / question / answer / post），用于 Deep Link Schema 选择 |
| title | VARCHAR(300) | 内容标题 |
| cover_url | VARCHAR(500) | 封面图链接 |
| original_url | VARCHAR(500) | 原文链接 |
| published_at | TIMESTAMP | 原始发布时间 |
| monitor_id | INT (FK) | 关联 monitors 表 |
| is_display | BOOLEAN | 是否在 H5 信息流中展示（默认 true；超 30 天软删除后为 false） |
| created_at | TIMESTAMP | 入库时间 |
| **UNIQUE** | `(platform, native_id)` | 全局去重唯一索引 |

### 7.4 假设（Assumptions）

| 编号 | 假设内容 | 验证方式 | 风险等级 |
|---|---|---|---|
| A1 | 抖音短链重定向后的 URL 中可以稳定提取 `sec_uid` | 用 10 个不同短链测试解析 | 中 |
| A2 | B 站 Cookie 有效期足够长（≥ 30 天），定期更新可维持 | 实测 Cookie 过期周期 | 低 |
| A3 | 知乎通过 RSSHub 中转可获取到个人主页/专栏的最新内容 | 部署 RSSHub 实例并测试 | 高 |
| A4 | Deep Link 在主流手机（iOS/Android）的浏览器中可正常唤醒 App | 在 5 款主流手机上实测 | 中 |
| A5 | YouTube 官方 Data API v3 的每日配额（10,000 units）足够个人使用 | 计算每小时请求量 | 低 |
| A6 | 通过 UserAgent 可准确识别微信/支付宝等受限环境，并正确分流跳转策略 | 在微信、支付宝、系统浏览器中分别实测 | 低 |

## 9. 非功能性需求

| 需求项 | 具体要求 | 衡量标准 |
|---|---|---|
| **低存储消耗** | 不存储正文/视频流，仅存文本和图片链接 | 平均单条 ≈ 1 KB；后端单条写入拦截上限 3 KB（防截断） |
| **数据生命周期** | 超 30 天内容软删除（is_display=false），数据保留 | H5 仅展示 is_display=true 的记录 |
| **异常容错** | Cookie 失效时 H5 继续展示历史缓存，不白屏 | 用户无感知降级 |
| **告警通知** | 连续 3 次抓取失败触发通知 | 调用企业微信/通知机器人 |
| **安全性** | Cookie、API Key 等敏感信息加密存储 | 不硬编码，使用环境变量或密钥管理 |
| **可扩展性** | 平台适配器采用插件化设计 | 新增平台只需添加一个适配器模块 |

### B.1 添加监控目标 — 完整业务流程

```

        │
        ▼
  ┌─ 前端校验 ─────────────────────────────────┐
  │  URL 是否为空？                              │
  │  ├── 空 → Toast "请输入链接"                 │
  │  └── 非空 → 正则校验 URL 格式                 │
  │       ├── 格式不合法 → Toast "链接格式不正确"  │
  │       └── 格式合法 → 调用后端 POST /api/monitors │
  └─────────────────────────────────────────────┘
        │
        ▼
  ┌─ 后端解析 ──────────────────────────────────────────────┐
  │  Step 1: URL 特征匹配，识别平台                           │
  │  ├── 包含 "douyin.com" 或 "v.douyin.com" → 抖音         │
  │  ├── 包含 "bilibili.com" 或 "space.bilibili.com" → B站  │
  │  ├── 包含 "zhihu.com" → 知乎                             │
  │  ├── 包含 "youtube.com" 或 "youtu.be" → YouTube         │
  │  └── 均不匹配 → 400 错误 "无法识别该平台"                │
  │                                                           │
  │  Step 2: 提取核心标识                                     │
  │  ├── 抖音: 短链重定向 → 解析最终 URL 中的 sec_uid         │
  │  ├── B站: 正则提取 /space/ 后的数字 → mid                 │
  │  ├── 知乎: 正则提取 /people/ 或 /column/ 后的 ID          │
  │  └── YouTube: 提取 @channel_name → 调 API 转 channelId   │
  │                                                           │
  │  Step 3: 去重校验                                         │
  │  ├── 查询 monitors 表 (platform + native_id)              │
  │  │   ├── 已存在 → 409 错误 "该博主已添加"                 │
  │  │   └── 不存在 → 继续                                    │
  │  │                                                        │
  │  Step 4: 获取博主昵称（异步，不阻塞添加流程）          │
  │  ├── 尝试调平台接口获取真实昵称                        │
  │  │   ├── 成功 → display_name = 真实昵称               │
  │  │   └── 失败 → display_name = "{平台名}_{id前8位}"   │
  │  │            （如"B站_mid12345"）                     │
  │  └── 标记 name_auto = true（系统自动获取，可被手动覆盖）│
  │                                                           │
  │  Step 5: 写入 monitors 表                                 │
  │  {platform, native_id, display_name, original_url,        │
  │   is_active: true, status: normal, fail_count: 0,         │
  │   last_content_at: now()}   // 初始值=当前时间，避免误触发停更提示 │
  │                                                           │
  │  → 返回 201 + monitor 对象                                │
  └───────────────────────────────────────────────────────────┘
        │
        ▼
  前端刷新列表，新博主出现在首行，Toast "添加成功"
```

### B.2 定时抓取 — 完整业务流程

```
Cron Job 触发（每 30 分钟一次）
        │
        ▼
  互斥锁检查
  ├── 上一轮仍在运行 → 记录日志，跳过本轮
  └── 获取锁成功 → 继续
        │
        ▼
  查询 monitors WHERE is_active = true
        │
        ▼
  按 platform 分组，组内串行遍历（同平台请求间隔 ≥ 1.5s）
        │
        ▼
  ┌─ 单条抓取流程 ─────────────────────────────────────┐
  │                                                      │
  │  Step 1: 根据 platform 选择适配器                    │
  │                                                      │
  │  Step 2: 适配器请求平台，获取前 6 条原始数据          │
  │  ├── 成功 → 返回原始数据数组                         │
  │  └── 失败 → 进入异常处理分支（见 B.3）               │
  │                                                      │
  │  Step 3: 数据清洗与标准化                            │
  │  对每条原始数据：                                    │
  │  ├── 提取 title → 清洗 HTML 标签                     │
  │  ├── 提取 cover_url → 统一为 HTTPS 绝对路径          │
  │  ├── 提取 original_url → 确保可直接访问              │
  │  ├── 提取 native_id → 平台内唯一内容标识             │
  │  ├── 提取 content_type → 判断内容类型                │
  │  │   (video / article / question / answer / post)    │
  │  └── 提取 published_at → 转换为 UTC 时间             │
  │                                                      │
  │  Step 4: UPSERT 去重与元数据同步                      │
  │  对每条标准化数据：                                  │
  │  ├── 查询 contents WHERE (platform, native_id)        │
  │  │   ├── 不存在 → INSERT 新记录，is_display = true   │
  │  │   └── 已存在 → UPDATE title/cover_url/original_url│
  │  │       ├── 若 is_display = true → 正常更新元数据   │
  │  │       └── 若 is_display = false → 仅更新标题和图片│
  │  │           严禁重置 is_display 和 created_at       │
  │  │           （防复活：旧数据不会被当作新内容展示）  │
  │  └── 统计本次新增条数（仅 INSERT 计入）              │
  │                                                      │
  │  Step 5: 写回前校验                                    │
  │  ├── 查询该 monitor 是否仍存在且 is_active = true    │
  │  │   ├── 已删除或已关闭 → 跳过写回，丢弃本轮结果     │
  │  │   └── 仍存在且开启 → 继续写回                     │
  │  │                                                    │
  │  Step 6: 状态回写（以请求成功为准，与新内容无关）     │
  │  只要适配器成功联通平台且未触发反爬拦截：             │
  │  ├── 强制更新 fail_count = 0, status = normal         │
  │  ├── 更新 last_sync_at = now()                        │
  │  ├── 有新增内容 → 额外更新 last_content_at = now()   │
  │  └── 无新增内容 → last_content_at 不变（追踪停更天数）│
  │                                                      │
  └──────────────────────────────────────────────────────┘
        │
        ▼
  遍历下一条 monitor...
  全部完成 → 本轮 Cron Job 结束
```

### B.3 异常处理与状态流转

#### Monitor 状态机

```
                    抓取成功
                   ┌──────────┐
                   │          ▼
              ┌────┴────┐  ┌──────┐
  首次添加 ──→│  normal  │  │ 恢复  │
              └────┬────┘  └───┬──┘
                   │           │
          失败1~2次│      抓取成功
                   ▼           │
              ┌─────────┐     │
              │ cookie   │────┘
              │ _expired │
              └────┬─────┘
                   │
          失败≥3次 │
                   ▼
              ┌──────────┐
              │ rate     │──→ 触发告警通知
              │ _limited │
              └────┬─────┘
                   │
              管理员修复后手动重置
                   │
                   ▼
              ┌──────────┐
              │  normal   │
              └───────────┘
```

#### 状态流转规则

| 当前状态 | 触发事件 | 目标状态 | 副作用 |
|---|---|---|---|
| normal | 单次请求失败（网络/反爬/鉴权） | cookie_expired | fail_count + 1 |
| cookie_expired | 请求成功（无论是否有新内容） | normal | fail_count 归零 |
| cookie_expired | 再次失败（fail_count ≥ 3） | rate_limited | 推送告警通知 |
| rate_limited | 请求成功（无论是否有新内容） | normal | fail_count 归零，告警解除 |
| rate_limited | 管理员手动重置 | normal | fail_count 归零 |
| 任意状态 | 管理员关闭监控 | 不变（Cron 跳过） | is_active = false |

#### 告警逻辑

```
抓取失败
    │
    ├── fail_count < 3 → 仅更新 fail_count，不告警
    │
    └── fail_count ≥ 3 → 检查告警条件
            │
            ├── 该博主 24h 内已告警过 → 跳过（静默期）
            │
            └── 未告警过 → 执行告警
                    │
                    ├── 更新 monitor.status = rate_limited
                    ├── 调用企业微信 Webhook / 通知机器人
                    │   消息模板：
                    │   "⚠️ [平台名] 博主 [昵称] 已连续 [N] 次抓取失败，
                    │    最后成功时间：[last_sync_at]，请检查 Cookie 或网络。"
                    └── 记录本次告警时间（用于静默期判断）
```

        ▼

  │  │                                                            │
  │  ├── 支付宝内置浏览器（UA 含 "AlipayClient"）                  │
  │  │   → 跳转策略 A：微信专用路径（同上，均无法唤醒外部 App）    │
  │  │                                                            │
  │  └── 系统浏览器（Safari / Chrome / 其他）                      │
  │      → 跳转策略 B：Deep Link 路径（见下方）                     │
  └────────────────────────────────────────────────────────────────┘
        │
        ▼

  ║  跳转策略 A：微信/支付宝内（受限环境）               ║
  ╚══════════════════════════════════════════════════════╝
        │
        ▼

        │

### B.5 H5 信息流 — 加载与筛选逻辑

```
页面加载
    │
    ├── 默认加载：GET /api/contents?page=1&size=20
    │   → 查询条件：is_display=true，返回全平台最新 20 条，按 published_at DESC
    │
    └── 用户点击平台 Tab（如"B站"）
        → GET /api/contents?page=1&size=20&platform=bilibili
        → 查询条件：is_display=true AND platform=bilibili

滚动到底部
    │
    └── 自动加载下一页：page++
        → 追加到列表尾部
        → 若返回空数组 → 显示"没有更多内容了"

下拉刷新（可选）
    │
    └── 重置 page=1，重新请求最新数据
        → 替换列表头部的新内容，保留未读部分
```

### B.6 边界场景与异常处理汇总

| 场景 | 处理方式 |
|---|---|
| 粘贴的 URL 已被删除/博主已注销 | 后端解析时返回 404，前端 Toast "该主页不存在或已失效" |
| 同一博主重复添加 | 后端去重校验 (platform + native_id)，返回 409 "该博主已添加" |
| B 站 Cookie 过期 | 适配器请求返回 401/403 → fail_count +1 → 状态流转为 cookie_expired |
| 知乎反爬触发验证码 | 适配器请求失败 → 降级尝试 RSSHub → 仍失败则 fail_count +1 |
| YouTube API 配额用尽 | 适配器请求返回 403 quotaExceeded → 本轮跳过，下轮 4 小时后重试 |
| 抖音代理 IP 被封 | 适配器请求超时 → 自动切换备用代理 IP 重试 1 次 |
| 微信/支付宝内点击卡片 | UA 检测为受限环境 → 跳过 Deep Link，直接弹出"复制链接 + 引导在浏览器打开"弹窗 |
| H5 首次加载无数据（新系统） | 信息流区域显示空状态："暂无内容，请先在管理端添加博主" |
| 数据库连接失败 | Cron Job 记录错误日志，跳过本轮，30 分钟后重试 |
| 封面图 URL 失效（404） | 前端 img onerror → 显示平台默认占位图 |
| contents 表 created_at 超 30 天的旧数据 | 每日凌晨 Cron 将 is_display 标记为 false（软删除），数据保留但 H5 不可见 |
| 上一轮 Cron 未完成，下一轮已触发 | 互斥锁检测 → 跳过本轮，记录日志，等下一个 30 分钟周期 |
| Cron 抓取期间管理员删除了某 Monitor | 写回前校验 monitor 是否存在 → 不存在则跳过写回，丢弃本轮结果 |
| 同平台多个 Monitor 密集请求 | 同平台请求间强制 sleep ≥ 1.5 秒，避免触发反爬频率限制 |
| 添加时博主昵称获取失败 | 降级为默认名称"{平台名}_{id前8位}"，Cron 下一轮抓取时异步补充 |
| 管理员手动修改了昵称 | 标记 name_auto = false，后续 Cron 不再自动覆盖该昵称 |
| 抓取到的内容 native_id 已存在 | UPSERT 机制：更新 title/cover_url/original_url，防止死链和破损图片 |
| UPSERT 命中 is_display=false 的软删除记录 | 仅更新标题和图片链接，严禁重置 is_display 和 created_at，防止旧数据复活 |
| YouTube monitor 距上次抓取不足 4 小时 | 本轮跳过该 YouTube monitor，等下一轮满足时间窗口后再执行 |
