## 1. 缺失的 PRD 业务项

### 1.1 数据库迁移（任务 1）

| 序号 | 缺失内容 | 风险等级 | 说明 |
|---|---|---|---|
| 1.1.1 | `monitors.native_type` 缺少 CHECK 约束 | **P0** | PRD 只写“仅在知乎场景下为 people / column，其余平台为空”，但任务计划没提加 `CHECK (native_type IS NULL OR native_type IN ('people', 'column'))`。没有约束，可能写入 `native_type='foo'` 导致 Cron 取错 RSSHub 路由。 |
| 1.1.2 | 知乎 `native_type` 非空约束缺失 | **P0** | `monitors` 表 `native_type` 为可空。如果知乎 monitor 添加时 `native_type` 漏写，Cron 的 `ZhihuAdapter` 无法判断用 `/people` 还是 `/column` 路由，会失败。任务计划没有通过 DB 约束或触发器强制“zhihu 必须带 native_type”。 |
| 1.1.3 | `platform_configs` 新平台配置行未定义 | **P0** | 任务 1 子任务只写“在 `supabase/seed.sql` 追加默认配置行”，但未明确新平台需要哪些 `config_key`。例如：知乎是否需要 `cookie` / `rsshub_route`；抖音/小红书是否需要 `proxy_list` / `rsshub_status` / `cookie`。 |
| 1.1.4 | `contents` 表未新增 `is_display=false` 相关索引 | **P1** | “已隐藏” Tab 查询 `is_display=false ORDER BY published_at DESC`。现有索引是 `idx_contents_platform_display`（platform, is_display, published_at），对“已隐藏” Tab（不带 platform 过滤）不走该索引。需要新增 `idx_contents_is_display_published`（is_display, published_at DESC）。 |
| 1.1.5 | 未迁移 `idx_contents_platform_display` 索引 | **P2** | 如果平台枚举变更后，PostgREST 的查询计划可能变化。无需删除，但需确认索引仍生效。 |

### 1.2 共享包 / Deep Link（任务 2）

| 序号 | 缺失内容 | 风险等级 | 说明 |
|---|---|---|---|
| 1.2.1 | `zhihu` + `post` 的 Deep Link 需要 `monitor_native_id`，但 content 查询未包含 | **P0** | PRD 中 `zhihu` + `post` 的 schema 是 `zhihu://people/{monitor_native_id}/pins/{native_id}`。当前 `contents` 表只有 `monitor_id`，Deep Link 生成需要额外 JOIN `monitors` 表获取 `native_id`，或者查询时把 `monitors.native_id` 带出来。任务计划没有明确 Deep Link 辅助函数如何拿到 monitor native_id。 |
| 1.2.2 | 小红书 `content_type` 映射未明确 | **P1** | PRD 写“图文笔记 -> post，视频笔记 -> video（或统一 post）”。任务计划任务 5 直接写“图文笔记 -> post，视频笔记 -> video”，但 PRD 括号里留了两个选项，任务计划没拍板。如果视频笔记也用 `post`，Deep Link 表里的 `video` 分支就永远不会命中。 |
| 1.2.3 | 未导出 TypeScript 类型扩展 | **P1** | 任务 2 只提到 `platforms.ts` 和 `deep-link.ts` 常量，没提到扩展 `Platform` 联合类型。`scripts/cron/src/adapters/types.ts` 和 H5/Admin 代码依赖该类型，必须同步更新 `packages/shared/src/index.ts` 导出。 |
| 1.2.4 | Admin 平台颜色未说明复用 shared 常量 | **P2** | 任务 10 写“新增平台 Tag 颜色”，但没明确是否从 `packages/shared/src/constants/platforms.ts` 读取。如果 Admin 单独写死颜色，后续改色会遗漏。 |
| 1.2.5 | 未新增 `getDeepLink` 单元测试的边界用例 | **P2** | 任务 2 写“单元测试覆盖所有 Deep Link 组合”，但 PRD 已说明抖音/小红书 scheme 需真机实测，单元测试无法覆盖“实测不可用”的 fallback 路径。需要增加 `original_url` fallback 用例。 |

### 1.3 Edge Function `parse-url`（任务 3）

| 序号 | 缺失内容 | 风险等级 | 说明 |
|---|---|---|---|
| 1.3.1 | 知乎专栏 URL 格式不明确 | **P0** | PRD 写 `zhuanlan.zhihu.com/xxx`，但知乎专栏真实 URL 通常是 `zhuanlan.zhihu.com/c/:column_id` 或 `www.zhihu.com/column/:column_id`；`zhuanlan.zhihu.com/p/:id` 是单篇文章。如果解析逻辑按 `zhuanlan.zhihu.com/xxx` 一刀切，会误把文章链接当专栏。任务计划需明确支持的精确 URL 模式。 |
| 1.3.2 | 小红书短链来源不明确 | **P1** | PRD 写 `xhslink.com/xxxxx`，但 `xhslink.com` 通常用于单篇笔记分享，不是用户主页。用户主页是否有独立短链域名？任务计划没有验证。如果用户只能拿到 `xhslink.com/笔记` 而不是 `xhslink.com/用户主页`，则解析后拿到的是 `note_id`，不是 `uid`，会整段逻辑出错。 |
| 1.3.3 | 短链解析缓存位置未定义 | **P1** | PRD 要求“将 short_code -> sec_uid 缓存 24 小时”。Edge Function 是无状态的，内存缓存无效。任务计划没有说明缓存介质：用 Supabase 表（如 `short_link_cache`）、KV、还是 Redis。 |
| 1.3.4 | 昵称降级格式与 1.0 SPEC 不统一 | **P2** | 1.0 SPEC 规定降级格式为“{平台中文名}_{native_id前8位}”。PRD 示例写“知乎用户_xxx”。任务计划没有明确最终采用哪种格式，前后端可能不一致。 |
| 1.3.5 | Edge Function 代理环境变量未明确 | **P2** | 任务 3 写“短链解析使用代理池（同 Cron 共享配置）”，但 Edge Function 的 env 注入机制与 Cron 不同（Supabase Function Secrets vs `.env` 文件）。任务计划没有说明如何在 Edge Function 中读取 `DOUYIN_PROXY_LIST`。 |
| 1.3.6 | 短链解析超时未定义 | **P2** | 短链重定向可能慢或卡死，任务计划没有指定 fetch timeout 和错误码。 |

## 2. 状态冲突

| 序号 | 冲突点 | 风险等级 | 说明 | 建议 |
|---|---|---|---|---|
| 2.1 | `is_display` 同时表示“用户隐藏”和“30 天软删除” | **P1** | 已按产品决策复用，但“已隐藏” Tab 会混进两类内容。用户可能找不到刚隐藏的内容。 | 在任务 9 的 UI 中明确告知用户“已隐藏列表包含手动隐藏和 30 天前自动过期的内容”；或 V1.2 拆分字段。 |
| 2.2 | `monitors.native_type` 可空但知乎必须依赖它 | **P0** | 如果知乎 monitor 的 `native_type` 为空，Cron 会失败。 | 在数据库层加 `CHECK (platform != 'zhihu' OR native_type IN ('people', 'column'))`；在 Admin 添加和 parse-url 返回时强制校验。 |
| 2.3 | `cookie_expired` 状态名对新平台语义偏差 | **P2** | 抖音、小红书、YouTube 都不是 Cookie 平台，但 `status` 仍有 `cookie_expired` 状态。 | 任务 7 中明确：新平台失败时同样进入 `cookie_expired`，状态名保留但语义为“抓取异常”。 |
| 2.4 | `zhihu` + `post` Deep Link 依赖 monitor 的 `native_id` | **P0** | 当前 `contents` 查询只返回 `monitor_id`，Deep Link 生成需要再查 `monitors.native_id`。 | 在任务 2 的 Deep Link 辅助函数中，要么要求调用方传入 `monitor_native_id`，要么在 H5 查询时 JOIN `monitors` 表。 |
| 2.5 | 短链缓存一致性 | **P2** | 抖音/小红书短链重定向目标可能变化（如用户改名后 UID 不变？），但缓存 24 小时可能返回旧的 sec_uid/uid。 | 任务 3 中定义缓存 key 为 `short_code`，并在解析失败时自动刷新。 |
