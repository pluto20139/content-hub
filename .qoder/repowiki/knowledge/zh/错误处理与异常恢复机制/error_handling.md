本项目采用**分层错误处理策略**，针对不同模块（Edge Functions、Cron 脚本、前端 H5）定义了明确的错误码、状态流转和兜底方案。核心目标是确保系统在第三方平台不稳定或网络异常时仍能保持基本可用性，并通过告警机制及时通知管理员。

### 1. Edge Functions 错误规范
Edge Functions（如 `parse-url`、`bilibili-auth`）遵循统一的 JSON 响应格式，通过 `success` 字段区分成败，并在失败时提供标准化的错误码。

- **统一响应结构**：
  ```json
  // 失败响应
  {
    "success": false,
    "error": {
      "code": "UNKNOWN_PLATFORM", // 标准化错误码
      "message": "无法识别该平台"   // 人类可读提示
    }
  }
  ```
- **核心错误码定义**：
  - `UNKNOWN_PLATFORM` (400): URL 特征不匹配任何已知平台。
  - `INVALID_URL` (400): URL 格式非法。
  - `DUPLICATE_MONITOR` (409): 尝试添加已存在的监控目标。
  - `BILIBILI_COOKIE_INVALID` (401): B站 Cookie 失效，需重新扫码。
  - `YOUTUBE_API_ERROR` / `RSSHUB_ERROR` (502): 下游服务调用失败。
  - `INTERNAL_ERROR` (500): 未预期的系统内部异常。

### 2. Cron 抓取引擎的容错与状态机
后端自动化引擎（GitHub Actions + Node.js）采用**状态机驱动**的错误处理逻辑，通过 `fail_count` 和 `status` 字段管理监控目标的健康度。

- **Monitor 状态流转**：
  - **Normal (🟢)**: 初始状态或抓取成功。成功时 `fail_count` 归零。
  - **Cookie Expired (🟡)**: 连续失败 1~2 次。通常由鉴权失效引起，系统尝试自动恢复或等待管理员干预。
  - **Rate Limited / Error (🔴)**: 连续失败 ≥3 次。触发**告警通知**（企业微信 Webhook），并进入 24 小时告警静默期。
- **并发与互斥保护**：
  - 使用 `pg_advisory_lock` 实现 Cron 互斥锁，防止多实例同时运行导致的数据竞争。
  - 同平台请求强制间隔 ≥1.5 秒，避免触发反爬频率限制。
- **写回前校验**：在更新 `monitors` 表状态前，校验记录是否仍有效（未被删除或关闭），防止无效写入。

### 3. 前端 H5 的降级与兜底策略
用户端 H5 强调**无感降级**，确保在 Deep Link 失败或环境受限时仍能提供基础浏览体验。

- **Deep Link 跳转兜底**：
  - **环境检测**：通过 UserAgent 识别微信/支付宝等受限环境，直接跳过 Deep Link，弹出“复制链接+引导浏览器打开”弹窗。
  - **超时检测**：在系统浏览器中，设置 2 秒超时计时器。若未监听到 `visibilitychange` 或 `pagehide` 事件，判定唤醒失败，弹出兜底弹窗提供网页版链接。
- **数据展示降级**：
  - 封面图加载失败时，自动替换为平台默认占位图。
  - 若数据库连接失败或无数据，展示空状态提示而非白屏。

### 4. 数据一致性与防复活保护
- **UPSERT 去重逻辑**：基于 `(platform, native_id)` 唯一索引。若命中已软删除（`is_display=false`）的记录，仅更新标题和封面，**严禁重置** `is_display` 状态，防止旧内容在信息流中“复活”。
- **软删除生命周期**：超过 30 天的内容自动标记为 `is_display=false`，H5 接口强制过滤此类数据，确保信息流时效性。

### 5. 告警与监控
- **告警触发条件**：单个 Monitor 连续抓取失败 ≥3 次。
- **告警渠道**：企业微信 Webhook。
- **静默机制**：同一博主 24 小时内不重复发送告警，避免消息轰炸。
